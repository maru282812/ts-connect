# WorkMarket 要件定義

最終更新日: 2026-04-01
対象リポジトリ: `c:\work\ts-connect`

## 1. 文書の目的

本書は、現時点の実装を基準に WorkMarket の要件を整理したものである。  
あわせて、現状不足している機能、仕様として曖昧な点、優先度の高い改善項目を明示する。

## 2. システム概要

- 社内または関係会社向けの案件マッチングサービス
- 一般ユーザーは案件閲覧、応募、問い合わせ、カジュアル投稿作成を行う
- 管理者は公式案件・カジュアル投稿の管理、応募一覧の閲覧、アーカイブ参照を行う
- 技術構成は Next.js App Router、Supabase Auth / PostgreSQL、HeroUI、Tailwind CSS、Resend

## 3. 利用者区分

### 3.1 一般ユーザー

- ログイン、サインアップ、ログアウトができる
- 公開中の投稿を閲覧できる
- 公式案件への応募、各投稿への問い合わせができる
- カジュアル投稿を作成できる
- 自分の応募履歴、自分の投稿、自分のプロフィールを確認できる

### 3.2 管理者

- 一般ユーザー機能相当の一部画面を `/company/*` 配下で利用できる
- 公式案件の作成・編集・状態変更ができる
- カジュアル投稿の作成・編集ができる
- 全投稿の管理、全応募の閲覧、アーカイブの閲覧ができる
- 管理者設定を確認できる

## 4. 現在の機能要件

### 4.1 認証・アカウント

- `/login` でメールアドレスとパスワードによるログインを行う
- `/signup` で一般ユーザーの新規登録を行う
- サインアップ時に以下を取得する
  - 表示名
  - メールアドレス
  - 会社名文字列
  - パスワード
- 認証後は `system_role` に応じて遷移先を分岐する
  - `ADMIN` は `/company/posts`
  - `USER` は `/app/posts`
- 未ログイン状態では `/app/*` と `/company/*` にアクセスできない
- ログアウト機能を提供する

### 4.2 権限・アクセス制御

- `users.system_role` に `ADMIN` / `USER` を持つ
- `middleware.ts` とレイアウトで画面アクセスを制御する
- Supabase RLS により以下を制御する
  - 一般ユーザーは公開中投稿のみ参照可能
  - 一般ユーザーは自分の応募のみ参照可能
  - 一般ユーザーは自分が作成者のカジュアル投稿のみ作成・更新可能
  - 管理者は全件参照・更新可能

### 4.3 投稿管理

- 投稿は `OFFICIAL` と `CASUAL` の 2 種類を持つ
- 投稿は少なくとも以下の項目を持つ
  - タイトル
  - 本文
  - 投稿種別
  - ステータス
  - 会社
  - 作成者
  - サムネイル URL
  - 掲載日時、更新日時
- 公式案件は追加で以下を持てる
  - 報酬テキスト
  - 担当者名
  - 締切日時
  - 募集人数
- 一部画面では応募上限表示を行う

### 4.4 一般ユーザー向け画面

#### 投稿閲覧

- `/app/posts`
  - 公式案件とカジュアル投稿を横断して一覧表示できる
  - タイトル・本文のキーワード検索ができる
  - `ALL` / `OFFICIAL` / `CASUAL` のタブで絞り込みできる
  - 投稿詳細ペインを同一画面内で表示できる
- `/app/official-posts`
  - 公式案件のみ一覧表示できる
- `/app/casual-posts`
  - カジュアル投稿のみ一覧表示できる
- `/app/official-posts/[postId]`
  - 公式案件詳細を表示できる
- `/app/casual-posts/[postId]`
  - カジュアル投稿詳細を表示できる
- `/app/posts/[id]`
  - 投稿詳細の互換ルートとして利用される

#### 応募・問い合わせ

- 投稿詳細から以下のアクションを実行できる
  - 公式案件への応募
  - カジュアル投稿への参加表明
  - 投稿への問い合わせ
- 応募・問い合わせ時はモーダルでメッセージを入力できる
- 同一ユーザーが同一投稿に対して同一種別の応募・問い合わせを重複送信できない
- 送信成功後は応募済み状態を UI に反映する
- `/app/applications`
  - 自分の応募一覧を確認できる
  - 投稿タイトル、応募種別、応募日時、ステータス、応募者メールを表示する

#### カジュアル投稿

- `/app/casual-posts/new`
  - 一般ユーザーがカジュアル投稿を新規作成できる
  - 作成時は自分の所属会社を `company_members` から解決する
- `/app/my-posts`
  - 自分が作成した投稿一覧を確認できる

#### マイページ

- `/app/settings`
  - 表示名を更新できる
  - 会社名文字列を更新できる
  - パスワードを変更できる
  - メールアドレスは参照のみ

### 4.5 管理者向け画面

#### 投稿管理

- `/company/posts`
  - 全投稿を一覧表示できる
  - 投稿タイプ別、進行状況別にタブ切り替えできる
  - クローズ済み投稿の表示切り替えができる
  - 投稿ごとの応募件数を参照できる
  - ステータス変更ができる
  - 編集画面への導線がある
- `/company/posts/new/official`
  - 公式案件を新規作成できる
- `/company/casual-posts/new`
  - 管理者が代理でカジュアル投稿を作成できる
- `/company/posts/[postId]/edit`
  - 投稿タイプに応じて編集 UI を切り替える

#### 応募管理

- `/company/applications`
  - 全応募を一覧表示できる
  - 応募者名、応募者メール、会社、応募種別、応募日時、ステータスを表示する

#### アーカイブ

- `/company/archive`
  - `CLOSED` の投稿を一覧表示できる
  - キーワード検索、投稿種別絞り込みができる
  - 詳細閲覧ができる
  - 複製して再作成できる
- `/company/archive/[postId]`
  - クローズ済み投稿の詳細を参照できる

#### 管理者設定

- `/company/settings`
  - 表示名を更新できる
  - 通知先メールアドレスの現在値を参照できる
  - 現在は環境変数ベースで表示のみ

### 4.6 管理者向けの一般ユーザー相当画面

- `/company/official-posts`
- `/company/my-posts`
- `/company/my-applications`
- `/company/my-page`

上記は管理者が一般ユーザー相当の導線を使うための画面として実装されている。

### 4.7 メール通知

- `/api/applications` で応募・問い合わせを受け付ける
- 応募登録後にメール通知を送信する
- 宛先は以下の優先順で解決する
  - 投稿作成者メール
  - `ADMIN_NOTIFICATION_EMAIL`
  - `admin@example.com`
- メール送信プロバイダは以下を切り替え可能
  - `console`
  - `resend`

### 4.8 データ要件

- `users`
  - ユーザー基本情報、システムロール、アカウント状態を保持する
- `companies`
  - 会社マスタを保持する
- `company_members`
  - ユーザーと会社の所属関係を保持する
- `posts`
  - 投稿本体を保持する
- `applications`
  - 応募・問い合わせ本体を保持する
  - 応募時点のスナップショットを保持する
    - 応募者名
    - 応募者メール
    - 応募者会社名
    - 投稿タイトル

### 4.9 互換ルート

- `/app/casual/new` は `/app/casual-posts/new` にリダイレクトする
- `/company/posts/new/casual` は `/company/casual-posts/new` にリダイレクトする
- `/company/posts/new` は現状 `/company/posts` にリダイレクトする

## 5. 現状不足している機能・改善が必要な機能

以下は優先度順に整理する。

### 5.1 最優先

#### A. DB スキーマと実装の不整合解消

- 実装では `OPEN` / `IN_PROGRESS` を使用しているが、初期マイグレーションの `posts.post_status` 制約は `DRAFT` / `PUBLISHED` / `CLOSED` のみ
- 実装では `requirements` を参照・保存しようとしているが、初期マイグレーションに該当カラムがない
- 管理画面でも「マイグレーション未適用環境を考慮して select を絞る」回避実装が存在する
- 要件として以下を明確化すべき
  - 正式な投稿ステータス集合
  - 公式案件の `requirements` カラム有無
  - 既存 seed / 型定義 / UI / RLS を含めた完全同期

#### B. 会社所属の登録フロー整備

- サインアップ時に会社名文字列は保存されるが、`companies` や `company_members` には自動登録されない
- 一般ユーザーのカジュアル投稿作成は `company_members` が存在しないと失敗する
- 要件として以下が必要
  - サインアップ時の会社紐付け方式
  - 既存会社選択か新規申請か
  - 承認フロー有無
  - 会社未所属ユーザーの扱い

#### C. 応募管理の実運用機能追加

- 管理者は応募一覧を閲覧できるが、ステータス更新、詳細確認、メモ管理ができない
- 一覧から `postId` で絞り込むリンクが張られているが、画面側でその条件が使われていない
- 要件として以下が必要
  - `APPLIED` から `REVIEWING` / `ACCEPTED` / `REJECTED` への更新
  - 投稿単位・応募種別単位の絞り込み
  - 応募メッセージ本文の閲覧
  - 対応履歴の保持

#### D. 応募上限の実制御

- UI では募集人数を表示しているが、応募 API では上限チェックを行っていない
- 要件として以下が必要
  - 上限到達時の応募拒否
  - キャンセル時の枠戻し
  - 応募順の採番と整合性

### 5.2 高優先

#### E. ロール判定の単一化

- 画面遷移やアクセス制御は `user_metadata.system_role` に依存する箇所が多い
- DB 側にも `users.system_role` があり、二重管理になっている
- 要件として以下が必要
  - 権限の正本をどちらに置くか
  - ロール変更時の同期方法
  - middleware / RLS / UI 判定の統一

#### F. 通知先メールアドレスの会社単位設定

- 現在の管理画面では通知先メールアドレスを表示するだけで保存できない
- 実装コメント上も会社ごとの通知先は将来対応扱い
- 要件として以下が必要
  - `companies.notification_email` などの永続項目
  - 会社単位の通知先編集 UI
  - 複数宛先対応可否

#### G. 投稿ライフサイクルの整理

- 一部画面は `PUBLISHED` を公開中と扱い、別画面は `OPEN` を公開中と扱っている
- カジュアル投稿と公式案件で状態遷移が揺れている
- 要件として以下が必要
  - 投稿種別ごとの状態遷移図
  - 公開中の定義
  - 公開終了、対応中、下書きの意味
  - 画面表示ラベルと DB 値の統一

#### H. 公式案件詳細項目の整理

- 公式案件フォームでは `requirements` を入力できるが、詳細画面での扱いが揃っていない
- 要件として以下が必要
  - 本文と募集条件の表示分離
  - 報酬、締切、担当者、サムネイルの必須/任意条件
  - 参考 URL、添付資料の要否

### 5.3 中優先

#### I. 会社管理機能

- 会社マスタの CRUD 画面がない
- 会社メンバーの追加・削除・権限変更画面がない
- 要件として以下が必要
  - 会社一覧
  - 会社作成
  - 所属メンバー管理
  - 会社ごとの管理者設定

#### J. 応募者・投稿者への追加通知

- 現状は投稿側への通知のみ
- 要件として以下が必要
  - 応募受付完了メール
  - ステータス変更通知
  - 問い合わせ返信通知

#### K. 検索・一覧 UX 改善

- 現状は基本的なキーワード検索のみ
- 要件として以下が必要
  - 会社別絞り込み
  - 締切日順、更新日順などのソート
  - ページネーション
  - 保存済み検索条件

#### L. 投稿の削除・復元

- 現状はクローズと複製はあるが、削除や復元の定義がない
- 要件として以下が必要
  - 論理削除の有無
  - アーカイブからの復元
  - 誤操作防止

#### M. 自己応募禁止などの業務ルール

- 投稿作成者本人が自身の投稿へ応募・問い合わせできる状態に見える
- 要件として以下が必要
  - 自社案件への応募可否
  - 自己投稿への応募可否
  - 管理者の代理応募可否

### 5.4 低優先だが必要

#### N. 入力バリデーション強化

- URL、文字数、数値、締切日時、メール文面に対する厳密検証が不足している
- 要件として以下が必要
  - フロントとサーバ両方のバリデーション
  - エラーメッセージ統一

#### O. 監査・運用要件

- 誰がいつ投稿や応募ステータスを変更したかの監査が残らない
- 要件として以下が必要
  - 監査ログ
  - エラー監視
  - メール送信結果の追跡

#### P. テスト整備

- 自動テストが存在しない
- 要件として以下が必要
  - 最低限の画面遷移テスト
  - 応募 API テスト
  - RLS / 権限制御テスト

## 6. 推奨する次の定義順

1. 投稿ステータスと DB スキーマの正式化
2. 会社所属フローの正式化
3. 応募管理ワークフローの正式化
4. 通知設計の正式化
5. 会社管理・一覧 UX の拡張

## 7. 補足

現状の実装は Phase 1 としては十分に骨格ができている。  
ただし、今後実運用に寄せるには「会社所属」「応募運用」「DB スキーマ整合」の 3 点を先に固めないと、追加実装を進めても仕様の手戻りが大きい。
